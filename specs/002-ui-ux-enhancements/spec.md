## 使用者情境與測試

### 使用者故事 1－語言切換需有明確回饋（優先度：P1）

使用者在切換語言時必須知道系統正在載入，避免困惑與重複點擊。

**優先原因**：這會影響所有頁面的體驗。若沒有載入提示，使用者可能以為網站壞掉並重複切換，造成多餘的 API 呼叫。

**獨立測試**：在任兩種語言與任一頁面間切換，確認載入遮罩會出現並在內容就緒後消失。此功能可立即改善體感效能並降低困惑。

**驗收情境**：

1. **假設** 使用者瀏覽任一頁面，**當** 切換語言時，**則** 頁面立即出現全畫面載入遮罩
2. **假設** 載入遮罩顯示中，**當** API 完成且新語言內容就緒時，**則** 遮罩消失並顯示目標語言
3. **假設** 使用者切換語言，**當** 載入遮罩仍在時，**則** 不會觸發重複的切換請求

---

### 使用者故事 2－透過推坑指南快速認識 KNK（優先度：P1）

新粉絲需要透過精選內容快速理解 KNK 的魅力、表演與綜藝亮點。

**優先原因**：推坑指南是主要轉換漏斗，呈現結構與內容的方式會直接影響互動與留存。

**獨立測試**：造訪推坑指南頁，確認各區塊依順序顯示且縮圖、內嵌影片正常，使用者可以不離開頁面即觀看推薦內容。

**驗收情境**：

1. **假設** 使用者開啟推坑指南，**當** 頁面載入時，**則** 「Why KNK」區塊位於最上方並提供吸引人的摘要
2. **假設** 使用者往下捲動，**當** 檢視內容時，**則** 區塊順序為：Why KNK、推薦舞台、推薦歌曲、推薦綜藝
3. **假設** 使用者看到內容卡片，**當** 檢視卡片時，**則** 卡片展示預覽縮圖
4. **假設** 使用者點開卡片，**當** 卡片展開時，**則** 卡片內顯示內嵌的 YouTube 播放器
5. **假設** 播放器已顯示，**當** 使用者播放影片時，**則** 影片在卡片內播放且不會開啟新頁籤

---

### 使用者故事 3－快速瀏覽綜藝內容（優先度：P2）

粉絲想快速查看綜藝出演，Prefer 直接進入內容，而非先挑選系列。

**優先原因**：簡化綜藝探索流程，對現有粉絲重要，但緊急性低於推坑指南。

**獨立測試**：造訪綜藝頁面確認所有內容以卡片呈現、不再分系列，並可一鍵開啟外部連結。

**驗收情境**：

1. **假設** 使用者造訪綜藝頁，**當** 頁面載入時，**則** 內容以單一卡片呈現、不含系列分類
2. **假設** 使用者看到綜藝卡片，**當** 檢視卡片時，**則** 卡片顯示縮圖與標題
3. **假設** 使用者點擊綜藝卡片，**當** 點擊後，**則** 內容在新分頁開啟

---

### 使用者故事 4－理解站點歸屬與內容權利（優先度：P3）

訪客需要知道誰製作網站、哪些內容由 AI 生成、媒體權利屬於誰，以建立信任並管理預期。

**優先原因**：重視法務與透明度，但對核心體驗影響較小，通常在閱讀完主要內容後才會看到 footer。

**獨立測試**：捲動至任何頁面底部，確認所需資訊存在。

**驗收情境**：

1. **假設** 使用者查看 footer，**當** 閱讀時，**則** 顯示「由小仙女 Pink 製作」或對應語言
2. **假設** 使用者閱讀 footer，**當** 檢視法務資訊時，**則** 免責聲明包含「影片、歌曲、歌詞、照片等由 220 Entertainment 所有」
3. **假設** 使用者閱讀 footer，**當** 審視內容標示時，**則** 提及「部分內容由 AI 生成」
4. **假設** 使用者閱讀 footer，**當** 看到錯誤回報資訊時，**則** 文字含有「聯絡我」等連至聯絡表單的連結

---

### 使用者故事 5－回報問題或提出需求（優先度：P2）

使用者需要分類清楚的表單來回報錯誤、委託網站製作或提出其他問題，並支援附件。

**優先原因**：有助於持續改善，但緊急性低於內容功能；透過表單回饋可提升品質。

**獨立測試**：造訪聯絡頁，使用各種詢問類型填單並送出，確認收到信件與提示。

**驗收情境**：

1. **假設** 使用者開啟聯絡頁，**當** 頁面載入時，**則** 表單顯示所有必填欄位
2. **假設** 使用者檢視表單，**當** 查看類型欄位時，**則** 下拉選單包含「錯誤回報」「網站製作委託」「其他」
3. **假設** 使用者填寫表單，**當** 輸入資訊時，**則** 表單包含：類型下拉選單、內容文字區、附件上傳
4. **假設** 使用者填妥資料，**當** 按下送出時，**則** 寄信到 pink.exp.studio@gmail.com
5. **假設** 信件送出，**當** 郵件抵達時，**則** 主旨格式為「knk-fans-site:{使用者選擇的類型}」
6. **假設** 使用者提交表單，**當** 完成送出時，**則** 看到成功訊息

---

### 使用者故事 6－站長掌握分析資料（優先度：P3）

站長需要了解訪客來源、行為與熱門內容，以利未來優化。

**優先原因**：對長期優化有價值，但不直接影響當前使用者體驗。

**獨立測試**：確認追蹤碼存在於所有頁面，並在 GA 後台看到測試流量。

**驗收情境**：

1. **假設** 任何訪客開啟頁面，**當** 頁面載入時，**則** 產生 page view 事件
2. **假設** 站長進入分析後台，**當** 查看訪客統計時，**則** 能看到總訪客數
3. **假設** 站長查看地理資訊，**當** 檢視報表時，**則** 顯示訪客所在國家/地區
4. **假設** 站長檢視熱門頁面，**當** 查看報表時，**則** 顯示最常被瀏覽的頁面
5. **假設** 使用者與工具互動，**當** 點擊互動元素時，**則** 事件被記錄

### 邊界情境

- 若語言 API 呼叫失敗？應顯示錯誤訊息並允許在原語言重試
- 若使用者快速連續切換語言？應取消待處理請求，只保留最後一次
- 若卡片中的 YouTube 無法載入？應顯示錯誤並提供直接開啟 YouTube 的連結
- 若聯絡表單附件超過 5MB？應顯示檔案過大的錯誤訊息並提示改傳較小檔案
- 若表單寄信失敗？應提示錯誤並重試或記錄以供人工處理
- 若缺少必填欄位？應阻止送出並標示未填欄位
- 若分析被瀏覽器/廣告阻擋？網站仍須正常運作，只是分析無法紀錄

## 需求

### 功能性需求

#### 語言切換載入體驗

- **FR-001**：切換語言時必須顯示全畫面載入遮罩
- **FR-002**：遮罩顯示期間必須阻擋頁面互動
- **FR-003**：所有語言資料載入完成後必須移除遮罩
- **FR-004**：載入期間不得接受新的語言切換請求

#### 推坑指南內容結構

- **FR-005**：推坑指南的第一個區塊必須是「Why KNK」
- **FR-006**：區塊順序必須為 Why KNK → 推薦舞台 → 推薦歌曲 → 推薦綜藝
- **FR-007**：每張內容卡片必須顯示縮圖
- **FR-008**：卡片需可展開顯示內嵌 YouTube 播放器
- **FR-009**：內嵌影片必須在卡片內播放，不得開啟外部頁面

#### 綜藝頁重新設計

- **FR-010**：綜藝頁必須以單一卡片呈現，不再分系列
- **FR-011**：每張卡片必須顯示縮圖與標題
- **FR-012**：點擊卡片必須在新分頁開啟內容

#### Footer 資訊

- **FR-013**：Footer 必須顯示「由小仙女 Pink 製作」或對應語言
- **FR-014**：Footer 必須包含「影片、歌曲、歌詞、照片等由 220 Entertainment 所有」
- **FR-015**：Footer 必須提及「部分內容由 AI 生成」
- **FR-016**：Footer 必須包含導向聯絡表單的連結文字（如「聯絡我」）
- **FR-017**：上述免責文字需支援所有語言（zh/en/ja/ko）

#### 聯絡表單

- **FR-018**：系統必須提供聯絡頁與表單
- **FR-019**：表單下拉必須包含「錯誤回報」「網站製作委託」「其他」
- **FR-020**：表單必須有訊息文字區
- **FR-021**：表單必須有附件上傳欄位
- **FR-022**：內文必須為必填
- **FR-023**：送出後必須寄信至 pink.exp.studio@gmail.com
- **FR-024**：信件主旨必須是「knk-fans-site:{使用者選擇的類型}」（例如：knk-fans-site:錯誤回報）
- **FR-025**：必須在送出前驗證必填欄位
- **FR-026**：送出成功後必須顯示確認訊息
- **FR-027**：附件需限制 5MB 內
- **FR-028**：送出失敗時必須顯示錯誤訊息

#### 分析整合

- **FR-029**：所有頁面必須追蹤 page view
- **FR-030**：必須追蹤訪客地理位置（國家/地區）
- **FR-031**：必須追蹤使用者對工具與互動元素的操作
- **FR-032**：若追蹤被阻擋，不得影響網站功能

### 主要實體

- **聯絡表單提交**：包含詢問類型（錯誤/委託/其他）、內文、附件、送出時間、使用語言等
- **語言載入狀態**：包含目標語言、待處理 API 清單、開始時間與完成狀態
- **推坑內容項目**：包含標題、描述、縮圖、YouTube ID、分類與排序
- **分析事件**：包含事件類型（page view/interaction/navigation）、時間戳、頁面位置、地理資訊與 session ID

## 假設

1. **Email 服務**：假設已有 SMTP 或郵件服務可寄送表單結果
2. **分析工具**：假設使用 Google Analytics 4（GA4）
3. **YouTube Embed**：假設嵌入 API 在目標地區可用
4. **檔案儲存**：附件直接附在郵件，不額外儲存
5. **檔案大小**：附件上限 5MB，足夠放螢幕截圖或文件
6. **語言 API**：沿用現行語系切換需呼叫 API 的機制
7. **載入遮罩設計**：樣式需符合現有設計系統
8. **表單在地化**：表單需支援四語介面
9. **錯誤記錄**：送出失敗需記錄以利除錯，細節未定
10. **效能目標**：語言切換在一般網路環境下應於 2 秒內完成
11. **分析隱私**：需遵守一般網路隱私規範（IP 匿名化、無 PII）

## 不在範圍

1. **Email 模板設計**：表單通知僅使用純文字 email
2. **表單防垃圾機制**：例如 CAPTCHA 之類功能暫不實作
3. **表單資料庫**：不將提交內容存入資料庫
4. **進階分析事件**：例如影片觀看時間或捲動深度
5. **客製化分析報表**：只使用 GA4 既有介面
6. **自動回覆信**：不主動寄信給使用者
7. **錯誤訊息翻譯**：瀏覽器預設驗證訊息即可
8. **行動 App 分析**：此專案僅針對 Web
9. **歷史資料匯入**：不從舊系統導入分析資料
10. **A/B 測試架構**：不建置新的測試平台

## 備註

- **安全性**：表單建議設置速率限制以防濫用，限制值未定
- **無障礙**：載入遮罩需具備 ARIA 標籤方便螢幕閱讀器辨識
- **效能**：YouTube 嵌入建議使用 lazy load 以降低初次載入成本
- **SEO**：推坑頁結構調整可能影響排名，若 SEO 重要可分階段釋出
- **分析法規**：須確認 GA 實作符合目標地區的隱私法（GDPR、CCPA 等）
- **未來增強**：後續可考慮為站長加入表單提交紀錄檢視介面
